## Problem Statement

BrandKit has implemented a coherent research prototype for spec-driven e-commerce creative generation, but it has not reached its stated Beta goal. `PRODUCT-PLAN.md` still labels the product as M0-A while M0-M3 reports claim completion, and neither document defines a Beta release gate. The next phase must prove that an individual or small team can install the project, configure a real brand, run repeatable campaigns, understand failures, and obtain useful outputs without repository-author intervention.

The present question is not whether BrandKit is production-grade. Production hardening is explicitly deferred until real use shows the approach is effective and reliable.

## What Makes This Cool

BrandKit's useful idea is not generic image or copy generation. It is a compiler-shaped workflow:

```text
Brand + product + channel + campaign specs
                  |
                  v
        deterministic compilation
                  |
          shared message plan
             /          \
            v            v
       visual output   content output
             \          /
              v        v
          deterministic checks
```

That structure makes creative changes inspectable and local: product facts and campaign intent are shared, while visuals and copy can regenerate independently. A useful Beta would let another team test whether this method reduces rework compared with ad hoc prompting.

## Constraints

- Current target: personal and small-team Beta, not a multi-tenant production service.
- Freeze new channels, templates, UI surfaces, databases, approval workflows, and collaboration features until Beta evidence exists.
- Preserve the current YAML + Python + shell CLI shape where practical.
- Existing user edits in `scripts/verify.py` must be retained.
- Beta must work from a clean clone and must not depend on repository-author context.
- External model calls remain optional for installation and offline CI. One declared provider is Beta-gated for online generation; missing credentials fail only when online mode is requested.
- First Beta support matrix: Ubuntu 24.04 and macOS 14, Python 3.11, Chromium installed by Playwright.

## Premises

1. The current repository is a research prototype with useful implemented flows, not a finished Beta.
2. Beta success means usable and reliable for an individual or small team; production-grade architecture is a later decision based on observed use.
3. The next milestone should freeze feature expansion and close installation, validation, reproducibility, failure visibility, and real-use evidence gaps.
4. M0-M3 reports are implementation history, not release readiness certificates.
5. `PRODUCT-PLAN.md` must become a living product baseline with explicit current status and Beta gates, rather than stating that its 2026-06-17 assumptions are no longer discussable.

## Approaches Considered

### Approach A: Usability Beta

Freeze scope and harden the existing flow around a small-team Beta contract. Add an explicit Beta milestone, machine-validated specs, clean-clone installation, campaign-isolated builds, actionable errors, a second real brand fixture, and task-level acceptance tests.

- Effort: medium; human team 2-3 weeks, AI-assisted implementation 4-6 focused days including review and CI stabilization.
- Risk: medium-low because it reuses the existing compiler and renderers.
- Reuses: YAML ownership model, compiler, shared message plan, CLI, renderers, validators, audit tests.
- Tradeoff: visible feature growth pauses while reliability and evidence catch up.

### Approach B: Research Evaluation Beta

Freeze all productization work and build a benchmark harness first: fixed prompts, reference campaigns, human scoring rubrics, cost/latency tracking, and pure-prompt comparisons.

- Effort: medium; human team about one week, AI-assisted implementation about one day.
- Risk: medium because evaluation quality depends on a credible scoring process.
- Reuses: baseline runner, reports, campaign fixtures, verifier outputs.
- Tradeoff: produces better research evidence but still leaves installation and team usability weak.

### Approach C: Feature-First Continuation

Continue adding templates, channels, and generation capabilities, then harden the accumulated system later.

- Effort: low per feature but unbounded in aggregate.
- Risk: high because every new surface multiplies unverified contracts and failure modes.
- Reuses: current extension points and templates.
- Tradeoff: demos improve quickly while Beta readiness moves farther away.

## Recommended Approach

Choose Approach A. It matches the stated target and makes the next decision reversible: after real personal/team use, the project can either deepen into a production system or remain a focused open-source research tool.

### Proposed Beta flow

```text
clean clone
   |
   v
documented install ---- failure ----> actionable setup error
   |
   v
validate all specs ----- failure ----> file + field + rule
   |
   v
compile campaign into isolated build directory
   |
   +--> offline deterministic render for CI/onboarding
   |
   +--> optional online generation with explicit provider errors
   |
   v
verify outputs + emit one campaign manifest
   |
   v
human acceptance rubric + repeat-task measurements
```

### Beta workstreams

1. **Plan and status reset**
   - Update `PRODUCT-PLAN.md` with an honest current-state table.
   - Add a dedicated Beta milestone after prototype M0-M3.
   - Reclassify milestone reports as historical evidence.

2. **Install and distribution contract**
   - Add a pinned dependency manifest and supported Python versions.
   - Make the CLI independent of the caller's current working directory.
   - Add clean-clone smoke tests for Ubuntu 24.04 and macOS 14 with Python 3.11.

3. **Spec contract**
   - Define `schema_version: 1` schemas for the six canonical inputs: brand core, visual spec, content spec, product facts, channel, and campaign.
   - Validate types, required fields, references, enums, and reject all unknown keys before compilation.
   - Replace the documented-but-not-implemented generic merge priority with explicit supported override rules.
   - Beta accepts only schema version 1; unsupported versions fail with the file, received version, and supported version. Migration tooling is deferred.

4. **Build isolation and failure contract**
   - Give each campaign run its own build directory and manifest.
   - Remove silent online-to-placeholder degradation unless the user explicitly opts in.
   - Standardize exit codes and structured diagnostics.
   - An actionable diagnostic contains an error code, input file, field or stage, human explanation, and a concrete repair action.
   - Campaign isolation permits repository code, selected templates, schema files, runtime dependencies, declared specs/assets, and the run's own build directory. It forbids reads from other campaign build directories and undeclared project data/spec/asset paths; an integration test runs two campaigns in both orders and concurrently.
   - Isolation fixtures include undeclared spec and asset files containing sentinel values; no sentinel may appear in outputs, manifests, logs, or traced file reads.

5. **Trustworthy verification**
   - Verify that evidence references exist and bind generated claims to known fact IDs.
   - Treat model-emitted provenance as an assertion to validate, not as proof.
   - Require generated copy to return ordered `segments`, each classified as `fact` or `subjective`; the final text is assembled only from validated segments.
   - A `fact` segment must carry a known fact ID plus an allowed `template_id`; the renderer fills fact values, units, and evidence references from Product Facts rather than accepting model-written factual text. Templates declare the fact types they accept, so an unrelated valid fact ID cannot satisfy the segment.
   - A `subjective` segment may contain model-written text but carries no fact ID and must not contain a number/unit, certification, ranking, comparison, absolute, or a configured factual product-property term.
   - Unknown segment types, unclassified free text, and subjective segments matching factual triggers fail closed. Fixtures cover representative numeric, certification, comparative, absolute, product-property, mood, and intended-experience cases.
   - Apply this deterministic classification only to the Beta content output types and templates; general semantic claim verification is deferred.

6. **Real-use validation**
   - Onboard one second brand without changing Python code. It must include licensed logo/product assets, at least three sourced product facts, both visual and content specs, and material visual/voice differences from Aether.
   - Run at least two campaigns per brand and the five modification tasks from the plan.
   - Record setup time, failures, model calls, non-target changes, human edits, and acceptance outcome.
   - Execute all five modification tasks against one declared second-brand baseline campaign. Save before/after artifact hashes, normalized manifests, provider call logs, verifier results, and human acceptance notes for every task.

7. **Online generation contract**
   - Gate Beta on SenseNova: copy model `deepseek-v4-flash` and background model `sensenova-u1-fast`, configured by `SENSENOVA_API_KEY`.
   - Credential absence, timeout, rate limit, malformed response, and provider error return distinct actionable diagnostics with no implicit placeholder fallback.
   - Record provider, model, latency, request attempt count, and response artifact hash in the campaign manifest without storing secrets.
   - Beta timeouts are 60 seconds for copy and 120 seconds for background, with zero automatic retries. Model identifiers are pinned for the Beta tag; changing either requires compatibility fixtures and release-note disclosure.
   - Copy responses must satisfy the validated segment schema; background responses must contain one decodable PNG payload. Any extra free text or malformed payload fails.

8. **Release engineering**
   - Starts only after workstreams 1-7 and every release gate pass.
   - Tag a GitHub prerelease and attach the compatibility matrix, changelog, known limitations, and evidence report.

|---|---|---|---|

### Human acceptance rubric

- Evaluate at least four final assets from the second brand: two visual and two content outputs.
- A non-author tester scores each asset from 1-5 for brand fit, factual correctness, channel fit, and practical usability with one written reason per score.
- Pass threshold: every dimension is at least 3, each asset average is at least 4, and no factual or compliance defect is accepted.
- Practical usability means the tester would use the asset after no more than one localized edit; required edits and elapsed time are recorded.

## Success Criteria

The Beta is releasable only when all criteria below pass. `Automated`, `Manual`, and `Mixed` identify the evidence type.

1. **Mixed:** A new user completes install and first offline campaign from a fresh checkout using only the README; CI reproduces the commands with empty dependency caches and network enabled only for dependency installation.
2. **Automated:** All six canonical spec types reject missing required fields, wrong types, all unknown fields, broken references, and unsupported schema versions with the diagnostic contract.
3. **Mixed:** Two materially different brands and at least four campaigns build without Python code changes; fixtures and the evidence report identify them.
4. **Automated:** Repeated identical offline inputs on the same OS/Python target produce byte-identical resolved task, message plan, content, HTML, and normalized manifest. PNG equality is checked by hash on the same target; cross-OS PNG identity is not required.
5. **Automated:** Ordered and concurrent campaign tests prove that no run reads another run's build directory or manifest, and sentinel fixtures prove undeclared specs/assets cannot affect outputs or be read by a run.
6. **Automated:** Missing assets, evidence files, templates, and browser dependencies fail with the diagnostic contract; missing provider credentials do so only when online mode is requested.
7. **Automated:** Every objective output claim uses a compatible factual template and valid known fact ID whose referenced evidence file exists; factual text is rendered from Product Facts, while ambiguous or invalid segments block the build.
8. **Mixed:** The five modification tasks satisfy the change boundaries and exact model-call budgets above.
9. **Manual:** At least one non-author tester passes the written acceptance rubric on the second-brand outputs.
10. **Automated:** CI runs unit, integration, clean-clone smoke, and offline end-to-end tests on Ubuntu 24.04 and macOS 14 with Python 3.11.
11. **Mixed:** Online generation passes one recorded happy path plus credential, timeout, rate-limit, malformed-response, and provider-error cases without implicit fallback.

The normalized manifest is canonical JSON with sorted keys and repository-relative paths. It excludes timestamps, absolute paths, run IDs, latency, request attempt counts, and other runtime observations; those remain in a separate non-reproducible execution record. Determinism compares only the canonical manifest and declared artifacts.

### Adjusted three-stage route

#### Stage 1 - Beta Contract and Red Tests

- Correct `PRODUCT-PLAN.md` status and demote M0-M3 reports to historical implementation notes.
- Add Beta gates and encode failing regression tests from the generated test plan.
- Require every declared target to produce a manifest entry and expected artifacts; zero-target and zero-check runs must fail.
- Make compiled channel YAML the sole validation contract and include factual claims inside visual templates in the provenance boundary.
- Secure the second brand and primary/backup non-author testers before setting a release date.
- Exit: tests demonstrate every P1 failure above; no implementation claims yet.

#### Stage 2 - Beta Foundation

- Add pinned installation/CI, schema v1, explicit override policy, root-independent paths, per-run directories/manifests, and structured diagnostics.
- Make offline mode hermetic and provider modes explicit.
- Scope all render/verify/validate work to one run manifest.
- Exit: clean-clone offline flow passes on Ubuntu 24.04 and macOS 14; concurrent campaign tests pass.

#### Stage 3 - Beta Evidence and Prerelease

- Implement fact-template binding, strict provider response contracts, and real evidence-file checks.
- Run two brands/four campaigns, five modification tasks, measured baseline, and non-author acceptance rubric.
- Optimize shared browser lifecycle and background cache only after correctness tests pass.
- Exit: all automated/manual Beta gates pass, then publish the GitHub prerelease.
